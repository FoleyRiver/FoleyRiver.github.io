BGP认证功能

原理概念

BGP是一种工作在AS之间的动态路由协议，具备强大的路径选择能力，这也能够让BGP协议能够管理超大型网络。对于超大型网络来说，路由的稳定性和安全性尤为重要。路由协议的报文一般都是明文发送的，如果网络攻击者伪造路由更新报文，或者篡改路由更新报文，就会造成严重的网络安全问题。因此我们会在部署各类路由协议时，会配置认证功能————就是指路由器对路由信息来源的可靠性及路由信息本身的完整性进行检测的机制。

BGP支持简单的密码认证方式，也支持安全性更高的MD5认证方式。
MD5认证方式：路由器会根据BGP报文的某些字段和密钥计算出一个128比特的散列值，然后将BGP报文联通散列值发送给邻居。邻居路由器在收到之后，会在本地基于接受到的BGP报文和相同的密钥再进行一次Hash计算。如果计算出的散列值一致，则认证通过，邻居关系则会正常建立；如果不同，则不通过，无法建立邻居关系，收到的BGP报文就会被丢弃。

实验内容

R1和R2属于同一个AS 100，而R3属于AS 200 ，R1于R2建立IBGP邻居，R2 与 R3建立EBGP邻居。为了保证BGP邻居之间发送的BGP路由信息的完整性，决定在BGP邻居之间配置认证功能。

实验拓扑

实验步骤

1.基础配置

首先我们将三台设备的邻居关系通过环回接口建立

[R1] bgp 100
    [R1] router-id 1.1.1.1
    [R1] peer 10.0.2.2 as-number 100
    [R1] peer 10.0.2.2 connect-interface LoopBack0

    [R2] bgp 100
    [R2] router-id 1.1.1.1
    [R2] peer 10.0.1.1 as-number 100
    [R2] peer 10.0.1.1 connect-interface LoopBack0
    [R2] peer 10.0.3.3 as-number 200
    [R2] peer 10.0.3.3 ebgp-max-hop 2
    [R2] peer 10.0.3.3 connect-interface LoopBack0

    [R3] bgp 200
    [R3] peer 10.0.2.2 as-number 100
    [R3] peer 10.0.2.2 ebgp-max-hop 2
    [R3] peer 10.0.2.2 connect-interface LoopBack0

R1与R2使用loopback0建立邻居关系时，需要保证可达性，我们可以使用别的IGP协议或者静态路由来保证底层的可达性。这里使用的是OSPF。

[R1] ospf 1 router-id 1.1.1.1
    [R1] area 0.0.0.0
    [R1] network 10.0.1.1 0.0.0.0
    [R1] network 10.0.12.0 0.0.0.255

    [R2] ospf 1 router-id 2.2.2.2
    [R2] area 0.0.0.0
    [R2] network 10.0.2.2 0.0.0.0
    [R2] network 10.0.12.0 0.0.0.255

R2与R3使用的是静态路由

[R2] ip route-static 10.0.3.3 255.255.255.255 10.0.23.3

    [R3] ip route-static 10.0.2.2 255.255.255.255 10.0.23.2

我们在R2上使用使用 Display bgp peer 来查看BGP邻居建立是否成功

[R2]dis bg peer 

    BGP local router ID : 2.2.2.2

    Local AS number : 100
    Total number of peers : 2                Peers in established state : 2

    Peer            V          AS  MsgRcvd  MsgSent  OutQ  Up/Down       State PrefRcv

    10.0.1.1        4         100       27       27     0 00:25:56 Established       0
    10.0.3.3        4         200       23       25     0 00:21:53 Established       0

可以看到R1、R2、R3之间的邻居状态均已Established状态，说明邻居关系已经正常建立。

2.配置基于单一密钥的BGP认证功能

经过上述步骤，BGP邻居关系已经建立成功，路由器之间已经可以相互传递BGP路由信息了。但是，没有配置BGP认证，所以网络会存在一定的安全风险。

接下来，分别在R1，R2，R3上配置BGP认证功能

[R1-bgp]peer 10.0.2.2 password simple BGP 

    [R2-bgp]peer 10.0.1.1 password simple BGP123
    [R2-bgp]peer 10.0.3.3 password simple BGP

    [R3-bgp]peer 10.0.2.2 password cipher BGP

我们在R2上使用使用 Display bgp peer 来查看BGP邻居状态

[R2]dis bg peer

    BGP local router ID : 2.2.2.2
    Local AS number : 100
    Total number of peers : 2                Peers in established state : 1

    Peer            V          AS  MsgRcvd  MsgSent  OutQ  Up/Down       State PrefRcv

    10.0.1.1        4         100        0        0     0 00:00:43     Connect       0
    10.0.3.3        4         200       45       48     0 00:43:39 Established       0

这里发现R2和R1的邻居状态是Connect,未能正常的建立起邻居关系，因为R1的所使用的密钥是“BGP”，而R2的密钥是“BGP123”，密钥不一致，无法正确的认证。

而R2和R3之间，即使双方使用密钥的方式不一样，R2选择的是Simple，R3选择的是Cipher，但密钥的内容都是BGP。所有是可以正常通过认证的。Simple和Cipher的区别在于，设备在存储密钥时，Simple是明文显示的，而Cipher是会被加密显示的。

R2:
        bgp 100
        router-id 2.2.2.2
        peer 10.0.1.1 as-number 100
        peer 10.0.1.1 connect-interface LoopBack0
        peer 10.0.1.1 password simple BGP123
        peer 10.0.3.3 as-number 200
        peer 10.0.3.3 ebgp-max-hop 2
        peer 10.0.3.3 connect-interface LoopBack0
        peer 10.0.3.3 password simple BGP

    R3:
        bgp 200
        router-id 3.3.3.3
        peer 10.0.2.2 as-number 100
        peer 10.0.2.2 ebgp-max-hop 2
        peer 10.0.2.2 connect-interface LoopBack0
        peer 10.0.2.2 password cipher $UhNHu.\NCZypQCee$t3U-U#

通过 display current-configuration查看R2和R3的配置文件我们能够发现通过simple方式的密钥内容 是正常显示的；而通过Cipher方式的密钥内容是密文显示的

3.配置基于Keychain 的BGP认证

上面的实验中，BGP进行认证时使用的是一个固定的密钥，当需要变换密钥来增强安全性时，操作将比较繁琐，也会造成BGP连接的中断。为此，我们可以使用基于Keychain的认证方式来实现密钥的周期性更换，对众多的密钥进行集中管理。使用Keychain的方式可以定义密钥的存活期，但应保证设备的系统时间一致，避免认证失败，可以使用NTP （Network Time Protocol）协议来保证时间的同步

Keychain方式下定义密钥的存活期分为Absolute与Periodic两种模式。Absolute模式下，密钥key的有效时间为一个绝对时间段；Periodic模式下，一个Key的有效时间为周期性的一段时间，分为Daily、Monthly、Weekly和Yearly等。以Daily为例，一个Key的有效时间为每一天的某一特定时间段。一个Keychain中可以有多个Key，最多支持64个key-id。

Key具有多个属性，包含Key-ID、认证算法、Key-string以及Send-time和Receive-Time，其中Send-time和Receive-time 用来定义Keychain中某个Key-ID的Active时间段。如果系统时间不在Send-Time或者Receive-Time时间内，则改Key-ID不会被使用。

接下来我们在R1和R2上配置基于keychain的认证公共，Key-ID为1，Key-String为BGP，选Periodic Daily模式。每天09：00-18：00使用Key-ID 1对发送的BGP报文做Hash运算，每天09：00-18：00使用Key-ID 1对接受到的BGP报文进行认证。

R1:
        keychain key mode periodic daily
        key-id 1
        algorithm md5
        key-string plain BGP
        send-time daily 09:00 to 18:00
        receive-time daily 09:00 to 18:00

        bgp 100
        peer 10.0.2.2 keychain key


    R2:
        keychain key mode periodic daily
        key-id 1
        algorithm md5
        key-string plain BGP
        send-time daily 09:00 to 18:00
        receive-time daily 09:00 to 18:00

        bgp 100
        peer 10.0.1.1 keychain key

我们在R2上使用使用 Display bgp peer 来查看BGP邻居状态，使用 display time-range all 来查看时间

[R2]dis time-range all
        Current time is 17:38:02-08:00 8-9-2022 Tuesday  

    [R2]dis bg peer

        BGP local router ID : 2.2.2.2
        Local AS number : 100
        Total number of peers : 2                Peers in established state : 1

        Peer            V          AS  MsgRcvd  MsgSent  OutQ  Up/Down       State PrefRcv

        10.0.1.1        4         100        0        0     0 00:00:43  Established       0
        10.0.3.3        4         200       45       48     0 00:43:39 Established       0

现在所在的时间是存在在sent-time 和 receive-time的范围内，所以能够正常的完成认证，并建立邻居关系。

在两台设备上使用keychain认证时，应保证Keychain的名称、Key-id、algorithm和Key-string保持一致。当一个Key的属性不完整或者系统时间不在定义的时间内，key处于Inactive状态。

在一个Keychain里，send-time时间不能重叠，但Receive-time可以重叠，保证在任何时间段内，BGP报文发送方只使用一个Key-id所对应的Key-string对发送的BGP报文进行Hash值计算，接收方对接收到的BGP报文将使用在receive-time为active的且Key-id相同的key-string来进行认证。如果认证设备之间的系统时间不一致，但使用的key-id相同，且自身设备上是Active的，认证也可以通过。

在R2上使用 display keychain key 命令来查看keychain的信息

[R2]dis keychain key

    Keychain Information:
    ---------------------
    Keychain Name             : key
    Timer Mode              : Daily periodic
    Receive Tolerance(min)  : 0 
    TCP Kind                : 254 
    TCP Algorithm IDs       :
        HMAC-MD5              : 5
        HMAC-SHA1-12          : 2
        HMAC-SHA1-20          : 6
        MD5                   : 3
        SHA1                  : 4
    Number of Key IDs         : 1
    Active Send Key ID        : 1
    Active Receive Key IDs    : 01
    Default send Key ID       : Not configured


    Key ID Information:
    -------------------
    Key ID                    : 1
    Key string              : BGP (plain)
    Algorithm               : MD5
    SEND TIMER              :
        Start time            : 09:00        
        End time              : 18:00
        Status                : Active
    RECEIVE TIMER           :
        Start time            : 09:00
        End time              : 18:00
        Status                : Active

显示信息表明了keychain的名称为key，密钥数量为1，status处于active状态。

在没有NTTP来保证时间同步的情况下，尽管管理员可以手动调整时间以尽量保证时间的一致性，但这样做精度很差。这种情况下可以使用 下面的命令来配置接受容忍时间，避免由于时间不同步或者key-id的变更过程中存在时间延迟而导致BGP报文认证失败的情况。接受容忍时间只对接受端的Key有效，其原理就是延长了Receive-Timer时间。

[R2-keychain]receive-tolerance infinite

infinite表示容忍所有时间延迟；也可以用某一具体时间代替，单位min，最大值是14400min。

为了避免在某一时刻没有活跃的key-id 而导致BGP 没有认证交互的情况，可以使用命令Default send-key-id 指定一个缺省的发送key-id。一个Keychain 中最多只能有一个key-id作为默认key-id。

当一个keychain中有多个key-id时，可以合理地给不同的key-id设置不用的send-time和receive-time，实现密钥的无缝隙周期性更换，并且不会导致BGP邻居关系的中断。当然如果密钥的更换存在时间缝隙，时间缝隙不超过180sBgp的链接也是不会中断的
